## Features

| File | Main Topics |
|------|-------------|
| **[auth.md](./auth.md)** | ‚ÄĘ **Passwordless**: magic link + OTP (no passwords)<br>‚ÄĘ Better Auth setup: `$lib/server/auth/index.ts` with Drizzle adapter<br>‚ÄĘ Client setup: `$lib/auth-client.ts` for browser use<br>‚ÄĘ Authentication flows: magic link, OTP code, OAuth (GitHub, Google)<br>‚ÄĘ Session management: database-backed, cookie-based, 30-day default<br>‚ÄĘ Route protection: per-route guards with `requireAuth()` helper<br>‚ÄĘ OAuth: `signIn.social()`, redirect handling, profile mapping<br>‚ÄĘ Email sending: magic link + OTP in same email, user chooses method<br>‚ÄĘ Hooks integration: populate `event.locals.user` and `event.locals.session`<br>‚ÄĘ **Rate limiting**: Better Auth's built-in is BROKEN ‚ÄĒ use sveltekit-rate-limiter<br>‚ÄĘ CSRF protection: enabled by default, `csrfProtection` option<br>‚ÄĘ Security: rate limits by endpoint, audit logging, 10-min token expiry |
| **[forms.md](./forms.md)** | ‚ÄĘ Hybrid approach: Superforms for most, Better Auth client for auth<br>‚ÄĘ Why Valibot over Zod: 1 KB vs 12 KB bundle, tree-shakeable, faster<br>‚ÄĘ Schema location: top of `+page.server.ts` (route-specific) or `$lib/schemas/*.ts` (shared)<br>‚ÄĘ Server integration: `superValidate()`, `valibot()` adapter, `fail()` for errors<br>‚ÄĘ Client integration: `superForm()`, real-time validation, debounce (300ms)<br>‚ÄĘ Validation timing: `oninput` (real-time), `onblur` (on blur), `submit-only`<br>‚ÄĘ Error display: inline (field-level), form message, toast notifications<br>‚ÄĘ Form patterns: settings (multi-section), wizard (multi-step), dependent fields<br>‚ÄĘ File uploads: `withFiles()`, Sharp processing, R2 storage, preview<br>‚ÄĘ Accessibility: labels, `aria-invalid`, `aria-describedby`, focus first error<br>‚ÄĘ Named actions: multiple forms per page<br>‚ÄĘ Confirmation modals: `onSubmit` cancel, programmatic `submit()` |
| **[state.md](./state.md)** | ‚ÄĘ Svelte 5 runes: `$state`, `$derived`, `$effect`, `$props`, `$bindable`<br>‚ÄĘ `$state`: reactive variables, deep proxy for objects/arrays<br>‚ÄĘ `$state.raw`: opt out of proxy, only reassignment triggers updates<br>‚ÄĘ `$state.snapshot`: plain object copy for API calls, cloning<br>‚ÄĘ `$derived`: computed values (memoized, lazy), use 90% of the time<br>‚ÄĘ `$derived.by`: multi-statement computations, must be pure<br>‚ÄĘ `$effect`: side effects after DOM update, cleanup pattern<br>‚ÄĘ `$effect.pre`: before DOM update (scroll position capture)<br>‚ÄĘ Module state: `.svelte.ts` files, factory pattern for instances<br>‚ÄĘ Context API: SSR-safe shared state, `setContext()` / `getContext()`<br>‚ÄĘ **Better Auth session state**: use `event.locals` (SSR-safe), not module-level `useSession()`<br>‚ÄĘ Reactive built-ins: `SvelteMap`, `SvelteSet`, `SvelteDate`, `SvelteURL`<br>‚ÄĘ UI state: sidebar (open/pinned), theme (light/dark/system), localStorage sync<br>‚ÄĘ Anti-patterns: modify state in `$derived`, use `$effect` for computation, forget cleanup |
| **[i18n.md](./i18n.md)** | ‚ÄĘ Library: sveltekit-i18n (lazy loading per locale, scales to 10+ languages)<br>‚ÄĘ Why not Paraglide: bundles all languages (multiplies by count), no runtime switching<br>‚ÄĘ Routing: `[[lang]]` optional parameter (`/`, `/de`, `/de/about`)<br>‚ÄĘ Translation files: `src/lib/i18n/{locale}/*.json`, per-route lazy loading<br>‚ÄĘ Route-based loading: `routes` array in loaders, only load needed translations<br>‚ÄĘ Usage: `$t('key.path', { params })`, pluralization, interpolation<br>‚ÄĘ Language switcher: dropdown, preserve current route, localized paths<br>‚ÄĘ Date/number formatting: native `Intl` API (locale-aware)<br>‚ÄĘ SEO: `lang` attribute, `hreflang` alternate links, translated meta tags<br>‚ÄĘ DB content: JSON columns for translations, helper function for fallback<br>‚ÄĘ Format helpers: `formatDate()`, `formatCurrency()`, `formatRelative()` |
| **[api.md](./api.md)** | ‚ÄĘ Strategy: REST (`+server.ts`) for external, Server Actions for forms<br>‚ÄĘ Route structure: `/api/*` for REST endpoints<br>‚ÄĘ HTTP methods: `GET`, `POST`, `PUT`, `DELETE`, `PATCH` exports<br>‚ÄĘ Request handling: `json()` helper, typed request bodies<br>‚ÄĘ Response patterns: `json()` with status, error handling, streaming<br>‚ÄĘ Validation: Valibot schemas for API payloads<br>‚ÄĘ Authentication: check `event.locals.user`, return 401 if missing<br>‚ÄĘ Rate limiting: per-endpoint limiters, stricter for mutations<br>‚ÄĘ Error responses: consistent format, status codes, error codes<br>‚ÄĘ CORS: headers for cross-origin requests, preflight handling<br>‚ÄĘ OpenAPI docs: schema generation, Scalar UI for interactive docs<br>‚ÄĘ Pagination: cursor-based for large datasets, limit + offset for simple<br>‚ÄĘ Webhooks: signature verification, idempotency, async processing |
| **[error-handling.md](./error-handling.md)** | ‚ÄĘ Error types: expected (`error()`), unexpected (`handleError`), form (`fail()`), API (`json()` + status)<br>‚ÄĘ Expected errors: use `error(status, { message })` helper in load functions<br>‚ÄĘ Status codes: 400 (bad request), 401 (not authenticated), 403 (not authorized), 404 (not found), 422 (validation failed), 429 (rate limited)<br>‚ÄĘ Error pages: `+error.svelte` global, layout-specific for `/app/*`<br>‚ÄĘ Custom error data: `App.Error` interface in `app.d.ts`<br>‚ÄĘ `handleError` hook: catch unexpected errors, log to Sentry, return safe message<br>‚ÄĘ Form errors: `fail()` in actions, Superforms integration, display patterns<br>‚ÄĘ API errors: `json({ error }, { status })`, consistent format with error codes<br>‚ÄĘ Database errors: catch PostgresError, map codes (23505 = unique violation, etc.)<br>‚ÄĘ File upload errors: size limits, type validation, 413/415 status codes<br>‚ÄĘ Client-side: custom `ApiError` class, `apiFetch()` helper, try/catch in components<br>‚ÄĘ AI errors: provider-specific (401/429/500), retry with exponential backoff, user-friendly messages<br>‚ÄĘ Testing: expect 404/403 for invalid access |
| **[rate-limiting.md](./rate-limiting.md)** | ‚ÄĘ **Critical**: Better Auth's built-in rate limiting is BROKEN (GitHub issues #2153, #2112, #1891)<br>‚ÄĘ Primary: sveltekit-rate-limiter (SvelteKit-native, serverless-compatible)<br>‚ÄĘ Production: Upstash Redis for distributed limiting across serverless<br>‚ÄĘ Strategy: global (hooks), route-specific (endpoints), Superforms integration<br>‚ÄĘ Rate limit types: IP (low resistance), IP+UA (medium), Cookie (high), User ID (highest)<br>‚ÄĘ Global limits: 100/min per IP, 50/30s per IP+UA, 200/min per cookie<br>‚ÄĘ Auth endpoints: 5/min (magic link/OTP), 3/hour (send code), 5/min (verify)<br>‚ÄĘ Time units: `s`, `m`, `h`, `d`, `15m`, `2h`, etc.<br>‚ÄĘ Hook integration: check in `handle`, skip static assets, return 429 with headers<br>‚ÄĘ Headers: `X-RateLimit-Limit`, `X-RateLimit-Remaining`, `X-RateLimit-Reset`, `Retry-After`<br>‚ÄĘ Per-user limits: custom rate function with `event.locals.user.id`<br>‚ÄĘ Upstash setup: `@upstash/ratelimit`, sliding window algorithm<br>‚ÄĘ Hybrid approach: in-memory (dev), Upstash (prod), environment-based switching<br>‚ÄĘ Recommended limits by endpoint type (login, API reads/writes, file uploads, contact forms) |
| **[progressive-revelation.md](./progressive-revelation.md)** | ‚ÄĘ **ProgRev**: Progressive content revelation based on user journey stage<br>‚ÄĘ Architecture: Server-authoritative (PostgreSQL) + client cache (localStorage)<br>‚ÄĘ Guest accounts: Created on first meaningful action, cookie-based identity (90 days)<br>‚ÄĘ FTUX phases: landing ‚Üí exploration ‚Üí commitment (pre-signup)<br>‚ÄĘ Guest-to-user merge: Atomic transaction on signup, PIONEER achievement granted<br>‚ÄĘ Database schema: guest, progrev_progress, onboarding_progress, achievements, bookmarks<br>‚ÄĘ Dual ownership: Two nullable FKs (guest_id XOR user_id) with CHECK constraint<br>‚ÄĘ SvelteKit hooks: Load guest identity, load ProgRev state ‚Üí event.locals<br>‚ÄĘ Components: ProgRevGate (conditional), ProgressIndicator (sidebar), AchievementToast<br>‚ÄĘ Form actions: recordAction() helper advances progress, invalidateAll() syncs<br>‚ÄĘ Cleanup: Daily cron deletes stale guests (>30 days, not merged)<br>‚ÄĘ Research validated: Progressive disclosure (NN/g), server-authoritative gating (Statsig) |
